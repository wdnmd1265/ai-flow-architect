## 一个 MD5 引发的血案

你让 GPT-4 设计一套用户认证系统。

它返回的代码看起来很干净——API 路由、数据库模型、中间件，一应俱全。你扫了一眼，没发现什么问题，上线了。

**你没注意到的是**：密码哈希用了 MD5，登录接口没有速率限制。

这不是段子，这是 AI 幻觉的真实杀伤力。它不是恶意，而是**一个模型没有任何机制能发现自己的盲区**。

类似的场景每天都在发生：

- AI 生成的代码看似正确，但隐藏安全漏洞
- AI 给出的方案看似完整，但遗漏了边界条件
- AI 的回答看似自信，但关键细节是编造的

**你信了，你就翻车了。**

## 问题本质：单模型自审 = 自欺欺人

现在的 AI 工作流框架怎么处理质量控制？

<!-- 图片位置：上传 comparison.png -->

| 方案 | 致命缺陷 |
|------|----------|
| **单模型反复提示** | 同一个模型审查自己的输出——盲区照旧 |
| **LangChain / CrewAI** | 编排很灵活，但质量把控是你的事 |
| **"相信 AI"** | 祈祷这次它蒙对了 |

核心问题：**同一个模型无法可靠地发现自己的错误**。

这不是 Prompt Engineering 能解决的。GPT-4 的训练数据决定了它容易在哪些地方犯错——而它恰恰在这些地方最"自信"。你让它自己检查，就像让一个色盲患者校对色彩。

---

## 解法：让两个 AI 互相打架

思路很简单：**如果一个模型看不到自己的盲区，那就换一个模型来看。**

<!-- 图片位置：上传 architecture.png -->

这就是 **audison** 的核心设计——双脑对抗架构：

```
你："设计一套用户管理系统"
       |
       v
┌────────────────────┐
│  Brain #1 规划脑    │  分析需求，生成执行蓝图
│  (GPT-4o)          │  标注风险点和替代方案
└────────┬───────────┘
         |
         v
┌────────────────────┐
│  对手脑 Opponent    │  从 5 个对抗视角挑刺：
│                    │  安全审计 / 成本分析 / 用户体验
│                    │  数据严谨 / 极简主义
└────────┬───────────┘
         |
    [你审批蓝图]
         |
         v
┌────────────────────┐
│  专家团队           │  创意专家 → 评估专家 → 编程专家 → 审核专家
│  (会话隔离)         │  每个专家独立运行，互不知晓
└────────┬───────────┘
         |
         v
┌────────────────────┐
│  Brain #2 仲裁脑    │  逐项比对蓝图 vs 交付物
│  (Claude)          │  不同模型 = 不同盲区 = 真正的对抗审查
└────────┬───────────┘
         |
         v
    你拿到的是：质量报告，不是赌博
```

**关键决策点全部留给人**——你审批蓝图、你决定是否通过、你拥有否决权。

AI 提议，你决定。

## 核心设计拆解

### 1. 对手脑：5 个视角的"杠精"

<!-- 图片位置：上传 opponent-brain.png -->

对手脑不是简单的"请检查一下"，而是从 5 个截然不同的视角发起挑战：

| 对抗视角 | 它在挑什么刺 | 举例 |
|---------|------------|------|
| **安全审计** | 安全漏洞、权限问题、注入风险 | "密码哈希算法未指定——请确认是否使用 bcrypt/argon2" |
| **成本分析** | 资源浪费、过度工程 | "这个方案用了 3 个微服务，单体架构就够了" |
| **用户体验** | 可用性问题、学习成本 | "登录流程需要 5 步，3 步就够了" |
| **数据严谨** | 数据完整性、一致性问题 | "用户状态枚举缺少'冻结'状态" |
| **极简主义** | 不必要的复杂度 | "为什么要自建缓存？Redis 就行" |

**在执行之前就拦截问题**——5 个视角在蓝图阶段就发起挑战，避免浪费一次 API 调用。

### 2. 跨模型仲裁：为什么要用不同的模型？

如果 Brain #1 和 Brain #2 用同一个模型，那就是**自审**——模型对自己犯的错最不敏感。

audison 强制模型隔离：

- Brain #1 用 GPT-4o → Brain #2 自动选择 Claude
- 只有一个 API Key？Brain #2 自动降级到同提供商的不同模型（如 GPT-4o → GPT-4o-mini）
- 不同的训练数据 → 不同的失败模式 → **真正的对抗审查**

```python
# 一个 Key 就能跑
architect = FlowArchitect(config={
    "brain1": "gpt-4o",
    # brain2 自动选 gpt-4o-mini
})

# 两个 Key 效果最强
architect = FlowArchitect(config={
    "brain1": "gpt-4o",
    "brain2": "claude-3-5-sonnet-20241022",
})
```

### 3. 三层 Prompt 栈

每个专家收到的 Prompt 由三层叠加：

```
┌──────────────────────────────────┐
│  Layer 3: 任务指令（每次不同）      │  当前步骤的具体指令
├──────────────────────────────────┤
│  Layer 2: 角色预设（按专家类型）    │  领域特定的行为规范
├──────────────────────────────────┤
│  Layer 1: 全局基座（硬编码）       │  "输出必须是合法 JSON，禁止废话"
└──────────────────────────────────┘
```

这保证了输出格式的一致性，同时允许每个专家有个性化的行为模式。

### 4. 会话隔离：专家之间不串通

每个专家运行在独立的 Session 中：

- 他们不知道彼此的存在
- 数据只通过结构化字段传递
- 没有"全量上下文倾倒"，没有信息过载

```python
class ProgrammerExpert(BaseExpert):
    required_input_fields = {"architecture", "api_spec", "data_model"}
    # 调度器只提取这 3 个字段传入，不传无关信息
```

## 实战：5 分钟跑通

### 安装

```bash
pip install audison
```

是的，已经发到 PyPI 了，一行搞定。

### 配置

```bash
# 创建 .env 文件
echo "OPENAI_API_KEY=sk-your-key" > .env

# 如果有 Anthropic Key（推荐，仲裁效果更强）
echo "ANTHROPIC_API_KEY=sk-ant-your-key" >> .env
```

### 运行

```python
import asyncio
from audison import FlowArchitect

async def main():
    architect = FlowArchitect(config={
        "brain1": "gpt-4o",
    })

    result = await architect.run("设计一套用户管理系统")

    if result["status"] == "success":
        print(f"质量评分: {result['audit_result'].get('score', 'N/A')}/100")
    else:
        print("质量检查未通过，建议修改：")
        for s in result.get("revision_suggestions", []):
            print(f"  - {s}")

asyncio.run(main())
```

## 与 LangChain / CrewAI 的本质区别

我不是要踩谁。这三个框架解决的是不同的问题：

| | audison | LangChain | CrewAI |
|---|---|---|---|
| **设计哲学** | 对抗式质量控制 | 灵活管道编排 | 角色驱动多智能体 |
| **质量把控** | 内置（双脑仲裁 + 对手脑） | 你自己负责 | 可选 |
| **模型隔离** | 自动强制 | 不强制 | 不强制 |
| **Token 节省** | 4 种机制，零配置 | 手动优化 | 手动优化 |
| **适用场景** | 你需要可信的输出 | 你需要最大灵活性 | 你需要多智能体模拟 |

**一句话**：如果你要最大灵活性，用 LangChain。如果你需要**可审计的质量保证**，这就是 audison 存在的理由。

## 省钱：4 种零配置省 Token 机制

AI 工作流最怕什么？Token 烧钱。audison 内置 4 种省钱机制，全部零配置开箱即用：

| 机制 | 怎么省 | 效果 |
|------|--------|------|
| **语义缓存** | 同一专家+任务命中缓存 → 跳过 API 调用 | 0 次调用 |
| **上下文压缩** | 历史超阈值 → 自动压缩 | 输入 Token 减少 ~60% |
| **本地规则预检** | 空任务/未知专家/无效复杂度 → 调用前拦截 | 0 成本 |
| **智能跳步** | 当前步骤失败 → 跳过剩余；低复杂度步骤自动跳过 | 0 次调用 |
