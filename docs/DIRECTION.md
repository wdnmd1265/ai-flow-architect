## 一、核心共识（无争议）

### 1.1 哲学定位：从"减少幻觉"到"让幻觉透明化"

**核心命题**：
- 幻觉是 LLM 的数学必然（中科院 AAAI 2026 证明），不是 bug
- 正确做法不是消灭幻觉，而是让幻觉透明化、可验证、人为可控
- 产品价值：让用户看到 AI 哪里自信、哪里分歧、哪里盲区

**一句话定位**：
> **"你不需要相信 AI，你只需要验证它。"**

### 1.2 法律体系类比（用于文档重写）

| 法律角色 | 对应模块 | 职责 |
|---------|---------|------|
| 检察官 | BrainOne | 提出主张，给出证据 |
| 辩护律师 | BrainOpponent | 攻击主张，找出漏洞 |
| 法官 | BrainTwo | 听取双方，做出裁决 |
| 上诉法院 | Formal Verifier（未来） | 用数学标准重新审查 |
| 庭审记录 | Evidence Chain DB | 不可篡改的完整记录 |
| 陪审团 | 跨模型仲裁团 | 多元视角验证 |

**这个类比比"三个大脑"更易理解**，应作为 Phase 3 README 重构的核心素材。

### 1.3 不做的事（三方共识）

| 不做 | 原因 |
|------|------|
| Lean 4 形式化验证 | 自然语言→形式化验证是未解决的科研问题，Axiom 团队做了多年。v3.0+ 愿景 |
| 砍到 3 个 Provider | 跨家族审查需要多样性，8 个 Provider 是资产不是负债 |
| 砍 fallback 降级路径 | 运维必需。OpenAI 挂 10 分钟系统不能停摆 10 分钟 |
| 压缩核心到 500 行 | 参照系选错。nanoGPT 750 行是教学玩具，12,700 行含测试、CLI、数据模型、配置管理是正常项目 |
| 改名 Aletheia | 现阶段机会成本太高，0 Star 阶段品牌重塑是自杀 |
| 砍 SDK / FastAPI | GitHub App 需要后端，自断后路 |
| 删除 FlowArchitect 代码 | 已验证的代码留着不碍事，零维护成本 |

## 二、Phase 3 规划（待执行）

### 2.1 核心方向：从"能用"到"值得信任"

**Phase 1 已完成**：核心功能（TrustEngine、三大脑、8 个 Provider 支持）
**Phase 2 进行中**：代码健壮性（--html Jinja2 报告、超时配置、异常处理、序列化测试、CLI E2E 测试、降级路径测试、trace_id）
**Phase 3 目标**：产品可信度——让用户第一次接触就知道"这个东西不是在骗我"

### 2.2 优先级排序

| 优先级 | 任务 | 产出 | 为什么是这个顺序 |
|--------|------|------|-----------------|
| **P0** | 信任地图 HTML 报告升级 | 争议热图 + 三色标注（置信/争议/盲区） | 用户第一印象，最大差异化 |
| **P0** | 证据链 SQLite 持久化 | 每次审查自动存库，可查询历史 | 护城河地基，越早积累越好。**无上下文终审依赖此项先行** |
| **P0.5** | 无上下文终审 | TrustEngine 流程新增一步：全新审查者不看前面结果 | ARIS 设计，防止锚定效应。**依赖 P0 证据链 DB 记录结果** |
| **P0.5** | 跨家族强制审查 | config 层 enforce：brain1 和 brain2 必须不同 provider | 已有 isolation_level，加一层硬约束 |
| **P2** | 4 层模型路由 | 本地初筛 → 工具锚定 → 跨模型审查 → 形式化验证（Lean 层留空） | 成本优化，但复杂度高，放后面 |

### 2.3 信任地图 HTML 报告设计

**当前**：sample-report.html 是扁平的列表
**升级方向**：锯齿状智能可视化

```
┌─────────────────────────────────────────┐
│  结论：第 37 条需要修改                  │
│                                         │
│  ████████░░  80% 置信（3/4 模型一致）    │
│                                         │
│  ① 赔偿责任上限过低  ██████████ 100%     │
│     ✓ 已验证（3 模型一致）               │
│  ② 管辖地条款不合理  ██████░░░░ 60%      │
│     ⚠ 争议（Claude vs GPT 分歧）        │
│     → 建议人工判断                       │
│  ③ 终止条件存在陷阱  ████░░░░░░ 40%      │
│     ✗ 无法验证                           │
│     → 仅供参考                            │
└─────────────────────────────────────────┘
```

**关键设计原则**：
- 不是"给你一个总分"，而是"把真相的全貌展开在你面前"
- 三色标注：置信绿 / 争议黄 / 盲区红
- 基于现有 TrustReport 的 `uncertainty` 和 `arbiter_votes` 字段
- **Phase 2 Jinja2 模板应预留争议热图插槽**

### 2.4 FlowArchitect 处理方案（折中）

**不砍代码，只砍曝光**：
- 代码保留，零维护成本
- README 里 FlowArchitect 已降到末尾作为"进阶路径"
- CLI 主入口已是 `ai-flow audit`，现状已满足目标，无需额外改动

**理由**：
1. 代码已经在那了，砍它省的是认知负荷不是代码量
2. 等证据链 DB 和 HTML 报告做出来，FlowArchitect 的价值自然被稀释
3. 现在砍是赌博，届时砍是清理

## 四、关键洞察与落地

### 4.1 哲学洞察落地

| 洞察 | 落地方式 | 时机 |
|------|---------|------|
| 法律体系类比 | 重写 README / 文档时用这个框架替代"三个大脑" | 下次文档更新 |
| 锯齿状智能可视化 | 渐进增强 --html 报告，三色标注 | Phase 3 |
| 证据链数据库 | SQLite 持久化每次审查的完整记录 | Phase 3 |
| "透明化"的关键 | 展示"没发现什么但不确定"比展示"发现了什么"更重要 | Phase 2 --html 报告改进 |

### 4.2 对 Phase 2 和 Phase 3 的影响

评审意见暗含的共识：**Phase 2 的 --html 报告是 Phase 3 的地基**

**Phase 2 已完成的工作为 Phase 3 打下基础**：
1. Jinja2 模板已实现，为争议热图插槽预留了扩展点
2. `--html` 中 `uncertainty` 的展示方式为 Phase 3 三色标注提供了数据基础
3. 当前 TrustReport 的 `uncertainty` 字段方向对，为"透明化"展示提供了数据支撑

**Phase 3 升级时需强化**：
**Phase 3 的 --html 不应该只输出**：
> "审查完毕，发现 X 个问题"

**而应该更突出地输出**：
> "审查完毕，以下区域我们无法确认是否有问题"

**虚假安全感（"审查通过了"）比没有审查更危险**。
